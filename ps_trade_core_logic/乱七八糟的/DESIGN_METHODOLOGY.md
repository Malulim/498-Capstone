# 从需求到代码：分层设计方法论

本文档的目标：给出一套**通用的、跟具体项目/语言无关**的方法，回答"拿到一个 code 任务，该按什么顺序、在哪些抽象层次上思考，最后才落到具体代码"。全文用本项目（AQTA 交易系统的 PS 核心逻辑）做例子，但每一条原则本身必须能原样套用到任何系统（例如一个汽车 ECU 模块）——凡是脱离了这个项目就说不通的话，本身就说明那条原则写错了。

**范围声明：** 本文档不覆盖"编译工具链怎么配置"（build）和"怎么调试"（debug），这两个话题另开文档。

---

## 0. 总览：层次是分辨率，不是步骤

下面列出的每一层，都是"用不同分辨率看同一个系统"，不是"做完第 1 步再做第 2 步"的流水线。层与层之间是**双向约束**：上层（规格）会往下压需求，下层（硬件现实）也会往上压限制，两头一起收紧，中间的设计空间才被确定下来。实践中你会在几层之间反复横跳，这是正常的，不是方法用错了。

| 层 | 回答的问题 | 产出物 | 在本项目中的例子 |
|---|---|---|---|
| 1. 规格层 | 必须为真的性质是什么？ | 一条条可验证的需求 | README 里的 FS2/FS3/FS4/FS12 |
| 2. 架构层 | 怎么切成互相通信的子系统？谁拥有什么？ | 模块清单 + 模块间契约 | Strategy Engine / Risk Guard / Config Loader / Execution Logger |
| 3. 算法层 | 某个模块内部，用什么计算过程解决问题？ | 一段可以脱离语言描述的过程 | "按 ID 查表选行为" vs "if/else 链" |
| 4. 数据结构层 | 撑起这个算法的表示形式是什么？ | 一个数据组织方式的选择 | ring buffer 存最近 N 个 mid price |
| 5. 语言层 | 这套算法+数据结构，在这门具体语言里怎么写？ | 实际源码 | struct / 函数指针 / `#include` |
| 6. 系统集成层 | 程序怎么真正碰到外部世界？ | I/O 调用点 | 读 JSON config、收发 UDP、访问寄存器 |
| 7. 部署/生命周期层 | 这个程序怎么变成一个活着、被监管的进程？ | 启动/隔离/保活策略 | isolcpus + `sched_setaffinity`、systemd/手动重启 |
| 8. 机器层 | 最终这些代码在硬件上怎么跑？ | 时序/内存预算 | 30μs 预算、766MHz、cache-DDR ring |

---

## 1. 规格层（Specification）

只描述"结果必须满足的性质"，完全不提怎么实现。好的规格层描述应该在任何语言、任何架构选择下都成立——如果一条"需求"里已经暗含了"用数组实现"这种细节，说明它其实混进了架构层的内容，需要拆开。

**判断标准：** 这条描述如果换一种完全不同的实现方式（比如从 C 换成 Python，从查表换成 if/else）还成立，它就是纯规格；如果不成立，说明写早了。

---

## 2. 架构层（Architecture）

这一层用一组**通用问题**把"要满足规格"的整体系统切成互相通信的模块。这组问题不针对任何具体领域，且**可以递归套用**——一个模块内部本身也是一个更小的系统，同一组问题可以缩小边界后再问一遍。

### 架构层的七个通用问题

1. **边界**：这个系统（或模块）的外部世界给它什么，它还给外部世界什么？——先切这一刀，再想内部。
2. **时间分层**：数据里，什么是每次循环都变的（tick 级）、什么是一次运行内固定但下次可能不同的（session 级配置，可运行时加载）、什么是除非重新设计/编译否则永远不变的（结构常量）？
3. **生命周期**：这个系统/模块有没有"一次性初始化"和"稳态重复运行"两个阶段？
4. **决策 vs 强制**：有没有"决定做什么"（可替换、可能出错的判断）和"这么做是否被允许"（无条件强制执行的守卫）这两个独立关注点？
5. **状态归属**：哪块随时间累积的内存，由哪一段代码独家负责修改？
6. **可观测性**：记录发生了什么，是不是该独立于"做这件事"的主线逻辑？
7. **可互换行为**：如果存在几种可互换的做法，怎么让调用者不需要知道具体是哪一种？（统一输入输出形状，是这个问题唯一的通用答案，语法上可以是函数指针表、也可以是字典、也可以是接口）

### 递归应用的例子（本项目）

**系统级**回答：市场快照进 → 订单决策出（Q1）；三层输入见上表（Q2）；Config Loader 是一次性阶段、Strategy Engine 主循环是稳态阶段（Q3）；Strategy Engine 是"决策"、Risk Guard 是无条件"强制"（Q4）；in-flight order table 由 Risk Guard 独家维护（Q5）；Execution Logger 独立于决策逻辑（Q6）；Momentum/Mean Reversion/Defensive 三种策略需要统一契约（Q7）。

**模块级**（把边界收缩到 Strategy Engine 内部）再问一遍同一组问题：进的是 Snapshot/RollingState/Params/position，出的是 Decision（Q1）；rolling state 逐 tick 变、params 是 session 级、RING_SIZE 是编译常量（Q2）；ring buffer 需要一次性初始化再进入逐 tick 更新（Q3）；active_strategy_id 决定用哪个策略（决策）、dispatch 机制本身跟选中谁无关（强制/机制）（Q4）；RollingState 这块内存该由谁独家写（Q5，落地方式见语言层的存储期限一节）；不需要模块内部单独日志（Q6，已由系统级 Execution Logger 覆盖）；三个策略函数统一签名（Q7）。

### 系统顶层的实例：程序入口的骨架

架构层不只用来切模块，它也直接决定"整个程序运行起来之后，代码该按什么顺序执行"——这个顺序就是 Q1/Q3/Q4/Q6 在系统最顶层被直接翻译出来的结果，跟用什么语言写无关，换成 Python/Rust 写出来形状是一样的：

```
程序入口:
    /* ── 阶段一：一次性初始化（Q3 的"一次性"半边）── */
    config = 加载并校验配置()                 /* 系统集成层：I/O，来源不限于文件 */
    若 配置无效: 报告致命错误(); 结束(失败)      /* 校验不过，绝不进入下面 */

    state = 初始化状态()                       /* Q5：这块状态该活多久、归谁管，决定它存储期限的落地方式 */
    应用部署层约束()                            /* 资源隔离等，具体机制因系统而异 */

    /* ── 阶段二：稳态循环（Q3 的"稳态"半边）── */
    当 应继续运行:
        input = 获取下一个输入()                 /* 系统集成层：I/O，来源不限于网络 */
        若 无新输入: 继续下一轮

        decision = 决策(input, state, config)              /* Q4：决策 */
        outcome  = 施加约束检查(decision, state, config)     /* Q4：强制，跟决策是谁无关 */
        若 outcome.被批准: 作用于外部世界(outcome)            /* 系统集成层：I/O，作用于外部世界 */

        记录(outcome)                                       /* Q6：可观测性，独立于决策主线 */
```

几点说明：

- 校验失败必须立刻结束，且排在任何状态初始化之前——这是"配置没校验过，不能开始处理外部输入"这条常见规格，在系统顶层唯一正确的落地位置。
- 程序入口本身不该包含任何模块的内部逻辑，只做编排：按 Q3 摆好两阶段顺序，稳态循环里按 Q1 定好的数据流依次调用各模块。如果这里出现了具体判断公式，说明算法层的内容泄漏进了本该只做编排的地方。
- "应继续运行"是留给"怎么优雅退出"的钩子，具体触发方式属于部署层细节，这里只需要知道循环需要一个可控的退出条件。

**本项目映射**：`加载并校验配置` = Config Loader；`决策` = Strategy Engine；`施加约束检查` = Runtime Risk Guard；`作用于外部世界` = 向 Exchange Simulator 发单；`记录` = Execution Logger；`应用部署层约束` = isolcpus 隔离下的绑核。

## 3. 算法层（Algorithm）

在某个模块内部，"用什么计算过程"解决它的契约，这一步**完全可以脱离任何语言描述**（伪代码或纯文字即可验证是否正确）。

**判断标准：** 同一个子问题，往往有不止一种算法能满足同一个架构层契约——"按 ID 选行为"既可以用 if/else 链，也可以用查表；两者都对，算法层要做的是从中选一个，理由通常来自下面的数据结构层或机器层反压上来的约束，而不是"哪个更优雅"。

**本项目例子：** Mean Reversion 用 SMA 算偏离、Momentum 用固定 lookback 算 delta——这些都是算法层的选择，写在 README 的批注里，属于"脱离 C 语法也能验证对不对"的内容。

---

## 4. 数据结构层（Data structure）

为算法层选定的过程，配一个具体的表示形式。**这一层的选择几乎总是被机器层（第 8 层）的约束逆向筛出来的，不是数据结构层自己"喜欢"哪种。**

**判断标准：** 问"如果没有第 8 层的约束（内存无限、延迟无所谓），这里还会选同一个数据结构吗？"——如果答案是"不会，会选更简单/更灵活的",说明这个选择是被下层约束逼出来的，值得在注释里写清楚"为什么"，而不是当成理所当然的默认项。

**本项目例子：** ring buffer 而不是动态数组，是因为机器层禁止热路径堆分配、且要求固定延迟——不是"C 喜欢数组"。

### 另一个维度：这块数据该活多久？

选表示形式（数组/链表/环形缓冲）只回答了"怎么组织数据"，还有一个独立的问题没回答："这块内存该在什么时候诞生、什么时候消失？"——这两个问题是正交的，一个数据结构可以配任意一种生命周期。

判断方法：问"这块数据的生命周期，是绑定在某一次函数调用上，还是绑定在整个程序的运行时间上？"如果是后者（比如一个要贯穿整个运行期、被反复读写的历史缓冲区），那么"只在某次函数调用期间存在"的内存从需求上就不是正确候选——哪怕技术上凑巧能跑（比如把它声明在一个恰好永不返回的函数里），那也只是"生命周期碰巧够长"，不是"设计上保证够长"，是脆弱的巧合，不是正确设计。这个判断本身跟语言无关，具体怎么在 C 里落地，见语言层。

## 5. 语言层（Language）

把算法+数据结构翻译成某具体语言的语法。**这一层内部本身有结构，且不同语言在这一层的规则天差地别**——C 靠文本 `#include` + 手动切 `.h`/`.c`，Python 靠 import 模块系统，没有头文件这个概念。语言层的通用性到此为止：往上四层（规格/架构/算法/数据结构）对任何语言都成立，语言层往下的具体规则必须按语言分别讨论。

### C 语言在语言层的具体规则

**唯一的第一性原理：** C 编译器把预处理展开后的文本从上到下单向扫过一次，一个名字必须在被使用之前，已经在它上面某处声明过。下面所有"该放哪一段"的规则都是这一条推出来的，不是风格偏好。

`.h` 文件内部顺序：
```
1. 头文件保护开始（#ifndef / #define）
2. #include 别的头文件（只包含本头文件自己要用到的类型所在的头）
3. #define 常量/宏（外部模块也需要用到的）
4. typedef 类型定义（struct/enum）
5. 函数原型（用到第4步定义的类型，所以必须排在其后）
6. 头文件保护结束（#endif）
```

`.c` 文件内部顺序：
```
1. #include 自己同名的 .h（拿到自己该实现哪些函数的承诺）
2. #include 别的实现需要、但接口不需要暴露的头
3. 本文件私有的 #define / 常量
4. static 的模块内全局状态（如果需要，见下）
5. static 辅助函数的原型（如果某辅助函数定义在调用它的函数之后，必须先声明）
6. 函数定义本体：公开的（对应 .h 原型）在前，私有的（static）在后，或按调用顺序排列
```

### 程序入口的 C 特有部分：签名与存储期限

架构层已经给出了程序入口该有的顺序（两阶段、数据流顺序），那是任何语言都一样的部分。C 语言在这之上，还有两件事是它特有的、必须在这一层单独决定：

**1. 入口函数的签名。** `int main(int argc, char *argv[])`——`argc`/`argv` 是部署层（谁启动了这个进程）跟语言层唯一的接口，命令行参数从这里进来。这不代表配置必须来自命令行参数，只是"外部启动方式"在 C 里落地的固定语法位置。

**2. 存储期限（storage duration）：这块内存该活多久、由谁拥有，在没有 malloc 的前提下该放哪。**

没有堆分配的嵌入式程序里，能用的存储期限只有两种，选哪个不是随便的，直接对应架构层的 Q3（这块状态该活多久）和 Q5（该被谁独家拥有）：

| 存储期限 | 生命周期 | 典型大小限制 | 适合放什么 |
|---|---|---|---|
| **automatic（栈）** | 函数返回就消失，作用域绑定 | 通常只有几 KB 到几十 KB（尤其是被隔离出来跑热路径的那个线程/核，栈大小往往是显式配置的） | 只在这一次调用内有用的临时值 |
| **static / global** | 编译期就定死地址，跟整个程序活得一样久，不需要谁释放 | 受限于总内存，不受"栈"这个概念限制 | 跨多次调用都要保留的状态，尤其是体积大的 |

**容易踩的坑：** 如果在程序入口里把一块长期状态声明成局部变量（比如 `State state = ...;`），因为入口函数本身不返回（程序一直在跑），这块栈内存"碰巧"活得跟整个程序一样久——但这是巧合，不是设计原则。真正该决定"这块内存放栈还是 static"的，是 Q3/Q5 的答案，不是"这个函数看起来活得久"这个偶然事实。对于体积大（比如一个几百 KB 甚至更大的定长缓冲区，塞不进典型嵌入式线程栈）、且希望"只有某一个模块能碰"的状态，正确做法是让它作为该模块内部的 `static` 变量存在——这样它既不占用任何函数的栈帧，也不会被程序入口意外看到或修改，Q5 的"独家拥有"由编译器/链接器强制，而不是靠"没人乱写"的君子协定。

**`static` 数据的三条硬性要求（不止"不是局部变量"）：**

1. **大小必须在编译期就已知。** 链接器要在编译时把这块内存的确切字节数定死，所以不能有"运行时才决定大小"的 static 数组——任何想做成可配置的尺寸，只能声明一个编译期上限，运行时只使用其中一部分，不能让 static 的维度本身依赖运行时加载的值（比如 config）。
2. **如果给了初始值，这个初始值本身也必须是编译期常量表达式。** 不能用函数调用的结果去初始化一个 static 变量——这是自动变量才有的自由度。没写初始值的 static 变量，C 标准保证自动清零，这是免费的好处；但如果初始状态不是"全零"，就必须显式写一个 init 函数在运行时赋值，不能指望初始化列表帮你算。
3. **它是全局唯一的一份内存，天然不区分"是谁在调用它"。** 如果不止一段代码会碰同一个 static 变量，没有同步机制就是数据竞争——这一点在单核单线程的程序里不成立（永远只有一个执行流碰它），但在多核/多线程系统里是真实约束：一个 static 变量的"独家拥有"承诺，只有在"确实只有一个执行上下文会碰它"这个前提成立时才有效，前提被打破（比如另一个 core 也想读它）就必须显式处理并发。

## 6. 系统集成层（Integration / I-O boundary）

程序编译出来之后，怎么真正跨出自己的内存空间，碰到外部世界。这一层的问题是"操作系统提供了哪些跟外部世界打交道的接口，该用哪一个"——跟语言层（怎么写语法）是完全不同的问题类别，即使代码表面上都是"调用一个函数"。

常见的几类 I/O 边界，每一类都有它自己的一套系统调用/机制，不能混为一谈：
- 文件 I/O（读 config：open/read）
- 网络 I/O（收发行情/订单：socket/recvfrom/sendto）
- 硬件寄存器访问（内存映射：mmap，或者本项目demo阶段用UDP代替的那部分）
- 人机交互（UART 控制台：读写字符设备）

**本项目例子：** Config Loader 读 JSON 属于文件 I/O；Data ingestion/Order output 在 demo 阶段是 UDP socket I/O（原设计是寄存器访问，因板子无网卡改用 UDP，见 README 3.2.2 的批注）。

---

## 7. 部署/生命周期层（Deployment）

编译好、也能做 I/O 的程序，怎么变成一个"活着、被监管、按需要保活"的进程。这一层要分清楚几件性质不同的事：
- **谁最初启动它**（手动 shell / 开机脚本 / 服务管理器）
- **崩了谁负责重启**（是否需要自动恢复，取决于规格层要求的可靠性程度）
- **资源隔离怎么落地**（内核启动参数决定"这个 core 不给调度器用"，是系统级、启动时一次性的设置；进程内部再用一个系统调用把自己绑定到那个 core 上——这是两个不同层级的动作，一个在内核配置里，一个在应用代码里，缺一不可）

**本项目例子：** isolcpus 是 PetaLinux 启动参数（内核级），`sched_setaffinity` 是进程内部调用（应用级）；demo 阶段用 tmux/手动重启，正式 24/7 生产级需求才需要 systemd 这类服务管理。

---

## 8. 机器层（Machine/hardware）

最终这些代码在硬件上怎么跑：时钟周期、内存布局、cache 行为。这一层不产出新代码，但它的约束会**逆向压回**第 3/4 层（算法/数据结构必须避开动态分配、必须是 O(1)），这是本文档反复强调的"双向约束"里最典型的一条路径。

**本项目例子：** 30μs 的 FS2 预算、766MHz 时钟、256MB 的 cache-DDR ring——这些数字本身就是压回上面几层的那把尺子。
